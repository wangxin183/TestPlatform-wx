# 你的任务
把每个自然语言步骤翻译成一个或多个可执行 DSL 步骤。补齐自动化执行必需但用例里缺失的信息：

1. 定位（locator）：为每个需要操作元素的动作给出定位。按平台选择类型：
   **iOS（XCUITest）** 可靠性优先：accessibility_id → name → predicate → class_chain → xpath → ocr_text
   - predicate 示例：`label == "登录"`
   - class_chain 示例：`**/XCUIElementTypeSearchField`
   **Android（UiAutomator2）** 可靠性优先：accessibility_id → id(resource-id) → text(精确文案) → uiautomator → xpath → ocr_text
   - text 示例：`{"type":"text","value":"推荐"}`（Tab/按钮可见中文）
   - uiautomator 示例：`{"type":"uiautomator","value":"new UiSelector().resourceId(\"com.iqiyi.acg:id/tab_item_text\")"}`
   - 输入框 xpath 示例：`//android.widget.EditText[1]`
   **禁止** 在 Android 脚本中使用 XCUIElementType、class_chain、predicate、name（iOS 专用）。
2. 数据（value）：input 动作若用例没给具体值，补一个合理的测试数据。
3. 断言分级：
   - 可判定正向（元素出现、文案匹配、页面跳转）→ 用 assert_visible / assert_text。
   - 负向「不出现/不展示/不应出现「文案」」→ 用 assert_text_absent（value=该文案）；禁止编成正向 assert_text。
   - 不可判定（"流畅无卡顿""图片逐步加载"这类主观/性能描述）→ 用 screenshot 留证，不要编造硬断言。
4. 模糊手势（"滑动到章节末尾""一直下滑"）→ 编译成带终止条件的 swipe：给 direction、ratio、times（重复上限），可配 until（出现某元素即停）。
5. 每个 DSL 步骤保留原始 NL 到 description，expected 保留用例的预期文案，便于报告展示。

# 平台动作工具 Schema（参数必须符合）
{action_tools_json}

# 动作参数规则
- tap / clear / assert_visible：必须有 locator
- input：必须有 locator 和 value
- assert_text：必须有 value（期望出现的文本），locator 可选（省略则全屏文本匹配）
- assert_text_absent：必须有 value（期望不出现的文本），locator 可选
- swipe：direction 必填，可选 ratio(默认0.6)/times(默认1)/until
- scroll：direction 必填，可选 until
- wait：可选 until(locator) 或 timeout(秒)
- 首步通常是 launch_app（若用例假定 App 已在前台可省略）
